# 课件总结

将课件/讲义转换为简洁的考试导向速查表，包含准确的页码引用、中英双语术语，以及仅保留对理解有实质帮助的图表。

**核心原则：** 将课件总结视为长文档的控制器工作流——先全局阅读，按课时或主题分段，为每段调度专注的阅读器，合并为速查表草稿，然后运行全新的覆盖率检查器，最后生成最终产物。

## 不可妥协的规则

- **默认输出 Markdown**（PDF 为可选，需 LaTeX）；产物直接放在源文件同目录，不创建子文件夹
- **触发此 skill = 必须产出文件**，聊天摘要不算成功交付（除非用户明确放弃）
- 中文为主，重要术语附原始英文；风格为考试导向的考点速查表（简洁但非压缩，单栏阅读）
- 每个知识点附 `[标题原文](源文件.ext#heading-slug)（pp. X-Y）` 格式的可点击页码引用
- 包含公式、关键概念、代表性例题和解题技巧；不生成泛泛的散文式章节总结
- 提取质量差到无法保留公式/页码 → **停止并询问**，不转述不确定内容
- OCR/提取文本/逐页内容仍视为课件总结任务，同样产出文件；不因上下文压力跳过后续章节

## Skill 边界

- 当平台支持时，仍可使用任务或子代理机制，但这是执行细节，**不**构成加载其他 skill 的理由。

## 必须达成的结果

此 skill 默认的成功结果为：

- 保存一个名为 `Summary - <stem>.md` 的 Markdown 文件，放在与源文件相同的目录中
- （可选）如果用户请求 PDF 输出，生成名为 `Summary - <stem>.tex` 的 LaTeX 源文件，放在同一目录
- （可选）如果用户请求 PDF 输出且 LaTeX 环境可用，生成名为 `Summary - <stem>.pdf` 的编译后 PDF 文件，放在同一目录
- 所有生成的文件直接放在与源文件相同的目录中，不创建子文件夹，除非用户明确请求其他位置
- 最终回复报告产物路径和遇到的任何阻碍

如果没有原始文件路径，输入仅为源文件衍生的文本、OCR、提取图片或逐页内容：

- 如果存在用户提供的讲座名称，以其作为命名基础
- 否则使用后备名称 `lecture-slides`
- 默认输出为当前工作目录下直接生成文件，不创建子文件夹，除非用户指定了其他位置

以下**不**算成功完成（除非用户明确请求）：

- 仅在聊天中的内联摘要
- 从未写入磁盘的摘要草稿
- 停留在分析、分段或合并笔记阶段

## 何时使用

**触发场景**：用户希望从课件/讲义/课堂笔记中获得考点总结、速查表、公式表、复习资料或考试 cheat sheet。

**核心触发词**：课件、slides、lecture PDF、考点总结、提炼考点、公式速查表、概念速查表、cheat sheet、考前整理、复习资料

**触发即产出文件**：`总结一下这个课件` 等请求 → 默认产出文件，不降级为聊天摘要。

**不适用于**：作业解答、论文写作、不含页码引用的普通文章摘要

## 输入格式

**推荐：Markdown (.md)**
- 最可靠的解析，无 OCR 错误
- 公式以 LaTeX 语法保留
- 用户可使用 Marker、Markitdown 或 Surya 等工具预先转换
- 如果用户提供 Markdown，直接跳到分段步骤

**支持：PDF (.pdf)**
- 可直接阅读，但识别可能有误
- 公式/图表可能丢失或误识别
- 对于扫描版 PDF，建议用户预先转换为 Markdown 以获得更好的质量
- 如果 PDF 读取失败，请用户提供提取文本或 Markdown

**支持：纯文本**
- 用户提供已提取的文本内容
- 视为相同工作流，跳过提取步骤

## 可移植性

- 这是一个独立 skill，无外部 skill 依赖。
- 如果子代理可用且实用，优先将其用于段落阅读器和独立的验证器。对于非常长的课件，当平台可靠支持时，这是首选执行路径。
- 如果子代理不可用，按顺序模拟相同的角色：控制器、逐段阅读、合并、验证。

## 工作流模型

- `控制器` 负责接收、全局阅读、分段、合并、样式规范化、构建选择和最终报告
- 每个 `段落阅读器` 仅负责一个课时块或主题块
- 一个全新的 `验证器` 在合并后检查覆盖率、页码引用、中英双语术语和考试实用性
- 控制器永远不要将段落笔记视为已可信的最终输出

## 调度规则

- **独立段落**（章节块、标题段落、独立页码范围）→ 平台支持时并行调度阅读器；**耦合段落**（推导链、跨页证明、依赖步骤）→ 顺序处理
- 每个阅读器只接收自己的页码范围 + 固定返回格式，不写最终文档
- 验证器必须是**全新上下文**，不复用阅读器；所有阅读器返回后控制器审查冲突→合并→交验证器

## 效率规则（运行时 Token 优化）

以下规则根据源文件 Markdown 行数（`wc -l`）自动选择工作流密度，避免小文档跑全量流程造成 token 浪费。

### 文档规模分级

| 规模 | 条件 | 工作流密度 |
|------|------|-----------|
| **小型** | ≤ 300 行 | 精简模式：跳过子 agent，全局阅读直接读全文 → 控制器自行生成草稿 → 自检 5 项清单 → 输出 |
| **中型** | 300–800 行 | 标准模式：首轮轻扫 + 2-3 段阅读器 → 合并 → 验证器抽查（5-8 个知识点 + 首尾段 + 公式密集段） |
| **大型** | > 800 行 | 完整模式：首轮轻扫 + 全部阅读器 + 全量子 agent 验证器 |

### 段落阅读器合并阈值

- 段落 < 80 行 → 与相邻段合并，减少子 agent 调度次数
- 子 agent 不可用时不强制模拟多个子 agent，控制器线性逐段阅读即可

### 验证器抽样模式

| 文档规模 | 抽查范围 | 验证不通过处理 |
|----------|---------|---------------|
| ≤ 300 行 | 控制器自检清单（覆盖/引用/双语/风格/例题，5 项打勾即通过） | 修复问题后直接输出 |
| 300–800 行 | 子 agent 抽查首段 + 尾段 + 公式密集段 + 5-8 个随机知识点 | 只修复问题段落，不重新全量验证 |
| > 800 行 | 全量核查 | 修复后再次全量验证 |

### 输出链接优化

- **同标题连续知识点共享链接**：段落内同一标题下的多个知识点，只在第一个附完整链接，后续知识点不再加任何"同属"标记（链接已足够定位，重复标注影响观感）
- **表格内链接简写**：表头已标注来源时，单元格省略链接只保留 `pp. X-Y`，表格下方统一注明 `*全部链接指向 [标题](file#slug)*`
- 删除源文件中的冗余过渡句、客套话、重复强调，输出只保留对考试直接有价值的要点

## 执行流程

1. 接收输入 → 确认源文件、输出格式和样式
2. 检测工具 → 确定最佳转换方式（详见步骤 1.5）
3. 全局阅读 → 建立分段地图，理解文档结构
4. 按课时/主题分段 → 保持主题完整性
5. 调度段落阅读器 → 逐段提取考点、公式、概念
6. 合并为速查表草稿 → 按主题组织，附页码链接
7. 🔴 调度全新验证器 → 检查覆盖率、引用格式、双语术语
   - 不通过 → 修复后重新验证
7.5. 🔴 调度格式检查器（全新子 agent，不复用验证器）→ 逐行扫描链接语法 + 符号完整性
   - 发现问题 → 修复后重新检查，直到零错误
8. 写入最终产物 → Markdown（默认），PDF（可选，需 LaTeX）

## 步骤 1. 接收输入

- 确认源文件路径（PDF、Markdown 或纯文本）。
- 如果用户已指定输出目录，确认之。
- 确定期望输出：`.md`（默认），`.tex` + `.pdf`（可选）。
- 确定用户需要整份课件还是部分。默认为整份。
- 记录用户已提供的任何样式指令。

除非用户另有说明，默认样式为：

- 语言：中文
- 排版：单栏正常阅读
- 用途：考试复习
- 语气：简洁、清晰、不啰嗦
- 输出：`Summary - <stem>.md`（默认）

## 步骤 1.5. 工具检测与工作流规划

**在阅读源文件之前，检测可用工具并规划最优工作流。**

### 检测清单

检查以下哪些工具可用（跳过读取工具）：

| 工具 | 检测方法 | 用途 |
|------|----------|------|
| **Marker** | `marker_single --help` 或 `where.exe marker` 或 `where marker` | PDF → Markdown（最佳质量，较慢） |
| **Markitdown** | `markitdown --version` 或 `where.exe markitdown` 或 `where markitdown` | 多格式 → Markdown（快速） |
| **Surya** | `surya --version` 或 `where.exe surya` 或 `where surya` | 通用 OCR |
| **Pandoc** | `pandoc --version` 或 `where.exe pandoc` 或 `where pandoc` | PPT/Word → Markdown |
| **LaTeX** | `xelatex --version` 或 `where.exe xelatex` 或 `where xelatex` | PDF 编译 |

### 工作流决策矩阵

根据检测到的工具和输入格式：

| 输入 | 条件 | 工作流 |
|------|------|--------|
| Markdown / 纯文本 | — | 跳过检测 → 直接步骤 2 |
| PDF | Marker 可用 | Marker → MD → 处理（最佳质量） |
| PDF | Markitdown 可用 | Markitdown → MD → 处理（快速） |
| PDF | Surya 可用 | Surya OCR → MD → 处理 |
| PDF | 均不可用 | 直接读 PDF（警告可能有误） |
| PPT | Pandoc 可用 | Pandoc → MD → 处理 |
| PPT | Markitdown 可用 | Markitdown → MD → 处理 |
| PPT | 均不可用 | 🔴 请用户手动转换 |
| 需 PDF 输出 | LaTeX 可用 | 处理 → .md + .tex + .pdf |
| 需 PDF 输出 | LaTeX 不可用 | 🔴 报告并询问用户 |

**🔴 检查点**：检测完成后报告可用/缺失工具，确认工作流后再继续。

## 步骤 2. 全局阅读（优先首轮轻扫）

不要直接跳到分块摘要。**默认采用首轮轻扫策略**，先读骨架再深读正文。

### 首轮轻扫（所有文档必做）

1. **有标题的源文件**：提取所有 `##`/`###` 标题行 + 每节首尾各 3 行正文 → 建立分段地图 + 主题列表 + 公式密集区标记
2. **无标题/结构不良的源文件**：按约每 150 行均匀切片 → 扫描每片首尾各 5 行正文 → 用首句关键词作虚拟标题 → 建立分段地图。标记"**无标题结构**"状态，后续链接降级处理（见[无标题源文件回退](#无标题源文件回退)）
3. 轻扫完成后正文留给段落阅读器按需深读

**例外**：小型文档（≤ 300 行）可跳过轻扫，直接读全文后进入输出（见[效率规则](#效率规则运行时-token-优化)）。

**阅读方式：**
- Markdown：直接阅读
- PDF：使用 Read 工具的 `pages` 参数（如 `pages: "1-5"`）
- 如果读取失败，请用户提供提取文本或 Markdown

- 如果在此步骤中将提取文本、OCR 输出或保存的笔记写入磁盘，必须写入源文件所在目录中。

对于大型文档（> 800 行），控制器必须在轻扫后产出：

- 总体主题列表
- 按行数/页码范围的分段边界
- 预期的课时/主题段落数
- 疑似公式密集或概念密集的区域
- 覆盖全部内容范围的分段地图

## 步骤 3. 课件分段

使用最粗粒度的分段，同时保持主题完整性。

优先级依次为：

1. 课件中已有的课时边界
2. 明确的标题边界
3. 主要概念过渡

除非必要，不要将紧密耦合的推导过程拆分到多个段落阅读器。

如果从课时标题或主题幻灯片无法明显看出段落边界，记录该分段的简短理由。

## 步骤 4. 段落阅读器阶段

如果子代理可用且实用，为每个独立段落调度一个全新的 `段落阅读器`。

如果子代理不可用，控制器仍必须在写入任何合并摘要之前创建单独的逐段笔记。不要用一次融合的通读来阅读和总结整份课件。

**段落合并**：段落 < 80 行 → 与相邻段合并，减少子 agent 调度次数（见效率规则）。子代理不可用时控制器线性逐段阅读，不强制模拟多个子代理。

如果提议的段落不够独立，无法安全地并行阅读，保持相同的逐段结构但顺序执行。

每个段落阅读器应恰好返回以下部分：

- `段落 ID`
- `覆盖的页码范围` — **必须同时包含两个维度**：
  - **原始页码**：原课件的幻灯片/页码（如 `物理页码 pp. 1468-1534`）
  - **所属标题**：该内容在源文件中隶属的最近 `##` 或 `###` 标题原文（如 `所属标题: ## CPU 时间公式及综合计算`），用于生成跨平台标题锚点
- `主要考点` — 每个考点后附最终格式的可点击链接：`[标题原文](源文件.md#slug)（pp. 页码）`
- `核心公式` — 每个公式后附同格式链接
- `附英文的关键概念` — 每个概念后附同格式链接
- `重要例题或常见题型` — 附同格式链接
- `解题技巧或易错点` — 附同格式链接
- `图表候选（如有）`
- `覆盖疑虑或不确定性`

段落阅读器不负责：写最终文档、决定排版、跳过页码引用、返回泛泛散文（详见反例清单）

## 步骤 5. 合并为速查表草稿

控制器将所有段落笔记合并为一个连贯的考试复习结构。

目标输出不是逐页叙述。应按课时或主题块组织，每个块下用简洁的要点列出。

每个主要块应尽量包含：

- 需要知道什么
- 公式
- 为什么重要或何时在题目中出现
- 一个代表性例题模式（如有用）
- 一个实用解题提示（如有用）

## 草稿样式规则

- 中文正文。
- 重要术语在中文术语后立即用括号附上英文。
- 简短要点，不要大段解释性段落。
- 偏好考试用语，如 `定义`、`公式`、`判据`、`技巧`、`典型例题`、`易错点`。
- 避免填充性过渡词和课堂讲故事式的表述。
- 除非定义必须精确，避免逐字复制幻灯片原文。
- 如果某个要点对速查表来说太详细，压缩到回忆方法所需的最少内容。
- 如果压缩到失去意义，略微展开而不是强行追求密度。

## 页码引用规则（可点击链接 = 标题锚点 + 页码）

**标准格式**：`[标题原文](源文件名.ext#heading-slug)（pp. X-Y）`
- 链接文字 = 源文件中该知识点所属的最近 `##` 或 `###` **标题原文**（未 slug 化的可读标题）
- 锚点 slug = 对标题文本应用下方 slug 规则（标题锚点跨平台通用：Obsidian/VS Code/GitHub/Typora 均支持；行号锚点 `#L<N>` 仅 VS Code/GitHub 支持，**禁用**）
- `（pp. X-Y）` = 原课件页码，作为辅助元数据跟在链接后面
- 总结文件与源文件同目录 → 相对路径直接用源文件名即可

### Slug 生成规则

取最近 `##`/`###` 标题全文 → ASCII 字母转小写 → 空格→`-` → 删除 `#` 前缀 → 中文/日/韩**原样保留** → 删除 `.` `,` `:` `;` `!` `?` `"` `'` `(` `)` `（` `）` → 合并连续连字符

| 源文件标题 | slug |
|-----------|------|
| `## CPU 时间公式及综合计算` | `#cpu-时间公式及综合计算` |
| `### CPI 的含义、影响因素、优化方向` | `#cpi-的含义影响因素优化方向` |
| `## Amdahl's Law` | `#amdahls-law` |
| `## 1.1 计算机发展史` | `#11-计算机发展史` |

**PDF 源文件**（仅未转换时使用）：`[标题原文](源文件名.pdf#page=1468)（pp. 1468-1534）`

### 输出链接优化（减少冗余）

- **同标题连续知识点共享链接**：段落内同一标题下的多个知识点，只在第一个附完整链接，后续知识点不加任何标记（不写"同属"、不重复链接、不加脚注）
  ```markdown
  - CPU 时间 = IC × CPI × T_c → [CPU 时间公式及综合计算](源文件.md#cpu-时间公式及综合计算)（pp. 1468-1534）
  - CPI 与频率无关
  - 减少任一项降低 CPU 时间
  ```
- **表格内链接简写**：表头已标注"来源"列时，单元格内可省略根链接只写 `pp. X-Y`；或统一在表格下方注明 `*所有链接指向 [标题](file#slug)*`
- 输出中删除源文件的冗余过渡句、客套话、重复强调，只保留对考试有直接价值的要点

### 无标题源文件回退
虚拟标题 = 首个关键概念名 → 链接降级为 `[概念名](源文件.ext)（pp. X）` + 在总结顶部加 `> ⚠️ 源文件不含章节标题，请按 pp. 页码手动定位。` + 向用户报告降级情况。

### 链接质量检查清单

写入最终文件前，全部打勾才能输出：
- [ ] 每个知识点引用是可点击 Markdown 链接（非纯文本 `pp. X`）
- [ ] 链接文字 = 标题原文（非页码），页码 `（pp. X-Y）` 跟在链接后面
- [ ] 每个链接有 `#heading-slug`（`.md`）或 `#page=N`（`.pdf`）
- [ ] slug 按规则正确转换，抽查 3-5 个确认源文件中标题存在
- [ ] 源文件名与实际完全一致（含大小写、中文）

### 常见错误（红线）

| 错误 | 问题 | 正确 |
|------|------|------|
| `pp. 1468-1534` | 纯文本，无法点击 | `[CPU 时间公式及综合计算](源文件.md#cpu-时间公式及综合计算)（pp. 1468-1534）` |
| `[pp. 1468](源文件.md#slug)` | 链接文字是页码，看不出跳转目标 | 链接文字 = 标题原文 |
| `[标题原文](源文件.md)` | 缺锚点，只打开文件无法定位 | 必须带 `#heading-slug` |
| `[标题](源文件.md#L1468)` | 行号锚点，Obsidian/Typora 不支持 | 标题锚点 `#heading-slug` |
| `[标题简称](源文件.md#slug-简称)（pp. X）` | 链接文字被缩写，与源文件标题不一致 | 链接文字 = 完整标题原文 |
| `[标题原文](源文件.md#CPU 时间公式)` | 锚点含空格/大写，解析可能失败 | slug = 全小写、空格改连字符 |
| `[标题原文](源文件.md#slug)` | 缺少页码元数据 | 链接后跟 `（pp. X-Y）` |

## 图表规则

仅包含对理解有实质帮助的图表（变换域/零极点/电路框图/时序图等），不添加装饰性图表。
- Markdown：Mermaid 或文字描述；LaTeX：`tikz`/`pgfplots`
- 图表简化到概念最低限度，支持具体要点
- 无法自信重建 → 停止询问，不编造

## 验证器阶段

草稿合并后，运行全新的验证器轮次。

如果子代理可用且实用，使用独立的全新 `验证器` 子代理。不要复用阅读器子代理作为验证器。

验证器必须检查（根据文档规模调整深度，见效率规则）：

- **覆盖率**：所有课时/主题段落均有体现，无跳过，分段地图每段在草稿中有对应（或明确说明省略）
- **页码引用**：格式为 `[标题原文](源文件.ext#heading-slug)（pp. X-Y）`；小型文档全量自检、中型抽查 5-8 个知识点 + 首尾段 + 公式密集段、大型全量核查
- **双语术语**：关键概念有英文，核心公式无遗漏
- **风格**：考试导向的考点速查表（非散文摘要），可读不压缩
- **例题与图表**：重要处包含代表性例题/解题模式，必要图表不遗漏且概念合理

验证器必须恰好返回以下部分：

- `裁决`，为以下之一：`通过`、`通过但有备注`、`不通过`
- `覆盖率发现`
- `样式和忠实度发现`
- `缺失或薄弱区域`

如果验证器报告缺失覆盖、页码引用薄弱、英文标签缺失、高价值例题缺失或源内容漂移问题，修复草稿并重新运行验证。

验证器不得在未进行直接源文件核查的情况下批准带页码引用的输出。

## 草稿关卡

**🛑 STOP · 必须满足所有条件才能继续**

在进入最终输出之前，必须满足：

- [ ] 所有预期段落均已处理
- [ ] 合并草稿已存在
- [ ] 页码引用对总结要点存在且连贯，并已格式化为指向源文件的可点击 Markdown 链接，**链接文字为标题原文、锚点为 `#heading-slug`（或 PDF 用 `#page=<num>`）、页码在括号中**，除非有明确的提取失败记录
- [ ] 中英双语术语已就位
- [ ] 验证器裁决为 `通过`
- [ ] 格式检查器裁决为 `通过`（链接语法、括号配对、符号完整性零错误）

如果验证器返回 `通过但有备注`，解决备注并运行全新的验证器轮次。不要在过期的验证器裁决上进入最终输出。

不要在草稿阶段声称任务完成。草稿完成不等于文档完成。

## 格式检查器阶段

内容验证器通过后，调度一个**全新的格式检查器子 agent**（不复用验证器），专门扫描链接和符号问题。内容验证器关注"写得对不对"，格式检查器关注"格式有没有坏"——两个关注点独立，不能合并。

格式检查器必须逐行扫描以下项目：

| 检查项 | 规则 | 典型错误 |
|--------|------|---------|
| **链接语法** | `[text](url)` 括号必须配对，`[]` 和 `()` 各自闭合；**链接文字必须是源文件中的真实标题**（不能是代码片段、公式、正文内容）；URL/锚点内的 `<` `>` 必须转义为 `\<` `\>` 或编码为 `%3C` `%3C`（否则解析器当 HTML 标签处理，打断链接匹配） | `[标题](file.md#slug` 少右括号、`[include](file.md#include <stdio.h>)` 内 `<` `>` 未转义导致链接断裂 |
| **锚点格式** | `#` 后为 slug 格式（小写、连字符分隔、无空格、无 `<` `>` `.` 等特殊字符）；**slug 必须对应源文件中实际存在的 `##`/`###` 标题**，不能从代码/公式/正文中生成 | `#CPU 时间` 含空格、`#1.1-计算机` 含句号、`#i n c l u d e <sp> < s t d i o .` 非法字符+非真实标题 |
| **页码格式** | `（pp. X-Y）` 使用全角括号，X ≤ Y | `(pp.1468-1534)` 用了半角、`（pp. 1534-1468）` 页码倒序 |
| **括号配对** | 全角 `（）` 和半角 `()` 各自内部配对 | `（pp. 1468-1534` 少右括号 |
| **符号完整性** | 无乱码、无截断的 LaTeX 公式、无悬空的 `$$` 或 `$` | `$$\frac{a}{b` 截断公式、`�` 乱码字符 |
| **列表格式** | `- ` 后有空格，嵌套缩进一致 | `-概念` 少空格、混合 tab 和空格 |
| **标题层级** | `#` 后有空格，层级不跳级（不从 `##` 直接到 `####`） | `##标题` 少空格 |
| **表格格式** | 表格前必须有**空行**（否则 Obsidian 不渲染）；列表项内的表格需**缩进到列表层级（2空格）且前有空行**；每行 `\|` 数量一致（含首尾）；分隔行每列至少 `---`；单元格内 `\|` 必须转义 | 表格紧跟标题/列表/段落无空行→Obsidian 显示源码、列表项内表格未缩进或无空行→不渲染、分隔行不足3连字符 |

格式检查器必须恰好返回：

- `裁决`：`通过` 或 `不通过`
- `错误清单`：每个错误的行号 + 错误类型 + 原文片段 + 修复建议
- `统计`：扫描总行数、链接总数、错误数

格式检查器 `不通过` → 按错误清单逐条修复 → 重新运行格式检查器 → 直到零错误。最多重试 2 轮；仍失败则报告给用户手动处理。

**🔴 检查点 · 格式检查器裁决必须为"通过"才能进入写入步骤。**

## 步骤 6. 写入 Markdown 输出

**默认输出格式：Markdown**

- 将合并草稿写入 `Summary - <stem>.md`，使用标准 Markdown 语法
- 数学公式：行内 `$...$`，独立 `$$...$$`；图表：Mermaid 或 `tikz`
- 页码引用格式：`[标题原文](源文件名.ext#heading-slug)（pp. X-Y）`，标题取自源文件最近 `##`/`###`
- 输出结构参考：`# 课程名·章节名` → `## 核心概念`（定义/公式/例题/易错点）→ `## 考试重点`（优先级表格）。完整示例见 [references/example-output.md](references/example-output.md)

## 步骤 7. PDF 输出（可选）

**PDF 输出为可选。** 仅在以下情况下继续：
- 用户明确请求 PDF 输出，且
- LaTeX 环境可用（TeXWorks、TeXLive、MiKTeX 等）

**如果用户需要 PDF 但没有 LaTeX 环境：**
1. 报告 PDF 输出需要 LaTeX 环境
2. 建议用户安装 TeXWorks 或其他 LaTeX 发行版
3. 不要悄悄跳过 PDF——询问用户如何处理

**如果 LaTeX 环境可用：**

- 将 LaTeX 源码写入 `Summary - <stem>.tex`
- **LaTeX 页码引用**：使用 `\href{源文件名.pdf\#page=1468}{CPU 时间公式及综合计算}（pp. 1468-1534）` 生成可点击链接，链接文字为标题名，页码附后（需要 `\usepackage{hyperref}`）
- 使用可用的 LaTeX 工具编译（TeXWorks、xelatex 等）
- 确认 `.tex` 和 `.pdf` 都已生成
- 将任何 LaTeX 致命错误或非零退出状态视为构建失败

**🔴 检查点 · 构建失败必须报告**
如果 LaTeX 编译失败：
1. 向用户报告错误
2. 不要悄悄降级为仅 Markdown
3. 等待用户决定

## 最终回复策略

- 不要用最终回复来替代缺失的产物。
- 除非文件确实已创建并检查，否则不要说工作已完成。
- 如果因缺少 LaTeX 环境、提取质量差或其他硬性前提条件而受阻，明确说明并停止，而不是悄悄降级为聊天输出。
- 完成后，最终回复应首先指向生成的产物路径，然后简要总结产出了什么。

## 输出命名

使用基于源文件名的确定性命名：

- Markdown 输出：`Summary - <stem>.md`，直接放在与源文件相同的目录中
- LaTeX 源码（可选）：`Summary - <stem>.tex`，放在同一目录
- 最终 PDF（可选）：`Summary - <stem>.pdf`，放在同一目录
- 任何辅助或中间生成的文件：放在同一目录中，命名为从 `<stem>` 派生的名称，而非 `output.*` 或 `temp.*` 等通用名称
- 不创建额外的子文件夹，除非用户明确要求

如果用户指定了自定义输出路径或文件名，遵循用户的指定。

如果不存在源文件路径，应用 `必须达成的结果` 中的后备命名规则，而不是临时发明新的命名方案。

## 反例清单（🚨 命中任一条 = 停止修复后再继续）

| # | 症状 | 后果 | 正确做法 |
|---|------|------|----------|
| 1 | **一次性总结整个长文档** | 覆盖偏移，遗漏后半 | 全局阅读 → 分段 → 逐段处理 → 合并 → 验证 |

| 4 | **链接缺标题锚点**（只 `(源文件.md)`） | 只打开文件无法定位内容 | 必须带 `#heading-slug` |
| 5 | **行号锚点 `#L<num>`** | Obsidian/Typora 不识别 | 统一用标题锚点 `#heading-slug` |
| 6 | **缺少英文术语** | 双语标注是核心交付物 | 重要术语后立即附英文括号 |
| 7 | **创建不必要的子文件夹** | 违背默认行为 | 直接放源文件同目录 |
| 8 | **跳过验证器步骤** | 可能遗漏内容 | 合并后必须运行独立验证器 |
| 9 | **跳过格式检查器** | 链接语法错误、括号不配对、乱码流出 | 验证器通过后必须运行格式检查器 |
| 10 | **编造不确定的数学公式** | 传播错误 | 质量差 → 停止并询问用户 |
| 11 | **强制要求 PDF 输出** | 用户可能无 LaTeX | 默认 Markdown，PDF 可选 |
| 12 | **跳过 MD 直接写 LaTeX** | Markdown 更通用 | 先 MD，再 LaTeX |
| 13 | **泛泛散文摘要而非考点** | 违背速查表定位 | 考点 + 公式 + 例题 + 技巧，简洁要点 |
| 14 | **文件未创建就声称完成** | 用户无交付物 | 文件实际写入磁盘后才报告 |
| 15 | **skill 触发后只给聊天摘要** | 未产出文件产物 | skill 触发 = 产出文件，除非用户明确放弃 |

| 17 | **丢弃例题/解题技巧** | 备考价值大打折扣 | 保留代表性例题模式和解题技巧 |
| 18 | **列表项内嵌表格不缩进/无空行** | Obsidian 不渲染，显示源码 | 列表项文字后加空行，表格缩进 2 空格到列表层级 |
